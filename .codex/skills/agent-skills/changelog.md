# Changelog のルール

## いつ更新するか

.git 管理されているリポジトリのコードを変更した場合、原則として changelog を更新してください。

### 更新不要なケース

以下の変更は changelog の更新不要です：

- GitHub Actions などのワークフロー設定
- 開発環境の設定変更
- その他、エンドユーザーに影響しない変更

## バージョン番号は記載しない

変更内容を追記する際、バージョン番号はリリース時に付与するため、記載しないでください。変更内容のみ箇条書きで追記してください。

## 記載対象ファイル

- WordPress プラグインの場合は `readme.txt` の `== Changelog ==` セクション。`readme.txt` が存在しない場合は `README.txt`、`readme.md`、`README.md` の順で存在するファイルを確認し、changelog が記載されていればそのファイルを更新してください。
- その他のプロジェクトでは `CHANGELOG.md` など、プロジェクトの規約に従ってください。

## 記載形式

各行の先頭に分類を括弧書きで記載してください。

- `[ 機能追加 ]`
- `[ 不具合修正 ]`
- `[ デザイン不具合修正 ]`
- `[ セキュリティ修正 ]`
- `[ 仕様変更 ]`
- `[ その他 ]`

例：
- `[ 機能追加 ] 投稿一覧にカテゴリーフィルターを追加`
- `[ 不具合修正 ] モバイル表示でメニューが崩れる問題を修正`

## 対象ブロックの記載

変更対象が特定の WordPress ブロックの場合は、分類の後にブロック名を `[ ブロック名 ]` の形式で記載してください。対象ブロックが Pro 版のみで提供されているものの場合は、ブロック名の後に `(Pro)` を付けてください。ただし、ブロック名自体に既に「Pro」が含まれている場合は `(Pro)` を追加しないでください。

例：
- `[ 不具合修正 ][ 投稿リストブロック (Pro) ] 投稿の取得件数が0になってしまう不具合を修正`
- `[ 機能追加 ][ カルーセルブロック ] スライドの自動再生オプションを追加`

## 記載言語

既存の changelog が全て英語で書かれているリポジトリの場合は英語で記載してください。
既存の changelog に日本語が含まれている場合は日本語で記載してください。

## 記載内容

- 一般ユーザーが読んでも何が変わったか分かるような表現にしてください。
- 内部実装の詳細（リファクタリング、変数名変更など）はユーザーに影響がない場合は省略して構いません。
